# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Go CLI (cobra) that downloads manga chapters from supported websites and packs them into CBZ files. Module: `github.com/elboletaire/manga-downloader`, Go 1.19.

## Commands

```bash
make install          # go mod download
make build            # clean + test + build unix binary (injects version via ldflags)
make build/all        # unix + windows binaries
make test             # go test -v ./... (uses richgo if installed)
go test ./ranges -run TestParse   # run a single test
make clean            # remove built binaries and *.cbz files
```

Unit test coverage is minimal (only `ranges/parser_test.go`). Real-world verification is done with the makefile smoke targets, which run the app against live sites:

```bash
make grabber          # all of the below
make grabber/inmanga
make grabber/mangadex
make grabber/mangabats
make grabber/tcb
make grabber/html     # runs a list of plainhtml-based sites
```

These hit live websites and download actual chapters, so run them selectively.

## Architecture

The download flow, all orchestrated from `cmd/root.go` (`Run`):

1. `grabber.NewSite(url, settings)` → `IdentifySite()` iterates the registered grabbers (`grabber/site.go`) and calls `Test()` on each, which fetches the URL and checks if the site matches. **Order matters**: `PlainHTML` is tried first, then site-specific grabbers (Inmanga, Mangadex, Tcb).
2. The matched `Site` fetches title and chapters (`FetchChapters` returns `Filterables`), the user's range argument (`ranges.Parse`, format `1-10,12,15-20`) filters them.
3. Chapters download concurrently (`downloader.FetchChapter`), bounded by `--concurrency` (chapters, max 5) and `--concurrency-pages` (max 10) semaphores. Pages are fetched via the `http` package (plain GETs with a Referer header).
4. `packer` writes CBZ files (`PackSingle` per chapter or `PackBundle` with `--bundle`), naming them via a Go text/template (`--filename-template`, parts in `packer/filename.go`). Duplicate filenames get a `v{{.Version}}` suffix instead of being overwritten.

### The grabber package

- `Site` interface (`grabber/site.go`) is the contract for all grabbers: `Test`, `FetchTitle`, `FetchChapters`, `FetchChapter`, etc. The base `Grabber` struct provides shared settings/helpers.
- `Filterable`/`Filterables` (`grabber/filterable.go`) abstracts chapters so they can be sorted/filtered by number; each grabber has its own chapter struct embedding `grabber.Chapter`.
- `PlainHTML` (`grabber/plainhtml.go`) is a generic goquery scraper driven by a list of `SiteSelector` entries (CSS selectors for title/rows/chapter/image). It covers mangabuddy, tcbonepiecechapters.com (TCB Scans), asurascans.com, demonicscans.org, mangakatana.com, rawkuma.net, zonatmo.org, mangapill.com, dynasty-scans.com, silentquill.net, templetoons.com, reader.deathtollscans.net (a FoOlSlide reader — series page is static HTML with a `.list .element` row per chapter, but the reader page only renders the current page's `<img class="open">`; every page's real URL is in a `var pages = [...]` JSON array with a `url` field, parsed via `getPlainHTMLImageURL`'s FoOlSlide branch — a pattern likely reusable for other FoolSlide-based scanlation sites), furyosociety.com, sanascans.com, witchscans.com, writerscans.com and mistscans.com. Selector order in the list matters because some sites have very similar markup.
- Site-specific grabbers with their own logic: `inmanga.go`, `mangadex.go` (both API-based, language-aware), `mangabats.go` (JSON chapters API + js image variables), `mangafire.go` (open JSON API: `/api/titles/{hid}` + paginated `/chapters` + `/api/chapters/{id}` for pages, language-aware), `mangak.go` (mangak.io, the mangabuddy rebrand: Next.js site, full chapter list and page-image URLs parsed from the `__NEXT_DATA__` JSON blob — the visible HTML only lists ~6 chapters, and the JSON `number` field is a 1-based ordinal so the real number is parsed from the chapter name), `qimanga.go` (paginated JSON chapters API + SSR reader), `tcb.go` (Madara-based wordpress sites, i.e. lhtranslation.net), `leercapitulo.go` (plain HTML chapter list, but the reader page only decodes its obfuscated page-image blob client-side; uses `browser.GetHTMLWithLocalStorage` to flip the site's own "load all pages" preference before scraping the resulting `<img>` tags — no cloudflare, no cipher reversing needed).

### The browser package

`browser/browser.go` drives a locally-installed Chromium browser (Chrome/Chromium/Brave/Edge, auto-discovered by chromedp's `ExecAllocator`) for sites plain HTTP can't scrape: JS-rendered pages, TLS-fingerprint blocks, or Cloudflare challenges. One shared browser process is started lazily and reused. `GetHTML(url, waitSelector, timeout)` returns rendered HTML and harvests the session cookies + real UA into the `http` package (`http/session.go`), so images still download via the fast plain-HTTP path reusing the browser's Cloudflare clearance.

`grabber/plainhtmlbrowser.go` (`PlainHTMLBrowser`) is the browser-rendered twin of `PlainHTML`: it reuses all the selector-driven parsing (`chapterFromDoc`), but matches sites **by domain** (a `browserSelectors` list of `BrowserSiteSelector`, no fetch — starting a browser is expensive) and fetches series/reader pages through Chrome. Registered first in `IdentifySite()`. Covers toongod, dragontea, kappabeast, sushiscan, toonily.

- **A wait selector matching on the first element isn't proof every element loaded.** mangahub.io's reader appends its page `<img>` tags progressively (not virtualized-by-scroll, just staggered over a few seconds); `GetHTML`'s default zero settle returned as soon as the *first* `img.PB0mN` appeared, silently truncating a 16-page chapter to 6. `BrowserSiteSelector.Settle` (wired through `browser.SetSettle`) adds a fixed extra wait after the selector first matches for exactly this case — verified by re-counting the selector matches with `PROBE_SLEEP` at increasing values until the count stopped growing, since a single probe run's count isn't trustworthy for pages that keep loading.
`grabber/plainhtmlbrowser.go` (`PlainHTMLBrowser`) is the browser-rendered twin of `PlainHTML`: it reuses all the selector-driven parsing (`chapterFromDoc`), but matches sites **by domain** (a `browserSelectors` list of `BrowserSiteSelector`, no fetch — starting a browser is expensive) and fetches series/reader pages through Chrome. Registered first in `IdentifySite()`. Covers toongod, dragontea, kappabeast, sushiscan, drakecomic.org.

- **Headless loses to Cloudflare; headed usually wins.** CDP automation is detectable, so CF challenges time out in headless mode. `browser.GetHTML` handles this automatically: it tries a short headless probe (`headlessProbeTimeout`) and, if the wait selector times out (a `challengeError`), tears down the headless browser and reopens a visible one (`goVisible`), where managed challenges resolve in a couple of seconds and the user can solve an interactive one manually. `--browser-visible` (`browser.SetVisible`) just forces headed from the start, skipping the wasted headless probe.
- **Always try the site's own API/HTTP first.** mangafire looks like it needs a browser (SPA behind CF), but its JSON API is wide open to plain HTTP — no browser needed. zonatmo's successor domain dropped the TLS block, so it's plain `PlainHTML` now. Only reach for `PlainHTMLBrowser` when HTTP genuinely can't get the data. silentquill.net (Armageddon Scanlation) uses the exact same mangastream/themesia markup as sushiscan (`#chapterlist li` / `.chapternum` / `ts_reader.run`), but has no Cloudflare in front of it, so it's a plain `PlainHTML` entry instead of a `PlainHTMLBrowser` one — same theme, different hosting, don't assume every themesia site needs a browser.

- **`tools/probe`** is the investigation tool: renders a URL in Chrome and tests selectors. Env knobs: `PROBE_VISIBLE=1` (headed), `PROBE_TIMEOUT=4m` (page-load timeout, default 45s — pair with a long `PROBE_SLEEP` to leave time for solving an interactive challenge manually), `PROBE_SLEEP=8s` (settle time for lazy SPAs), `PROBE_DUMP=/tmp/x.html` (dump HTML), `PROBE_NETLOG=1` (log non-asset network responses — how the mangafire `/api/chapters/{id}` endpoint was found), `PROBE_FETCH_SEL="sel"` (grab first matching image and test a plain-HTTP download with the harvested session), `PROBE_API_URL="url1,url2"` (after the browser clears a challenge, fetch arbitrary JSON API URLs via plain HTTP reusing the harvested session — checks whether cookies carry over to another host, e.g. an `api.` subdomain; pair with `PROBE_API_DUMP_DIR=/tmp/x` to save full bodies instead of a truncated preview).

The wall is that each page's image is decrypted client-side (`window.YggdrasilDecipher.decipher`, their "AetherCipher") into a `blob:` `<img>` **only on genuine, dwelt human interaction** — after a fresh navigate the active page never decodes on its own (40s+ dwell → no `<img>`), and every automated arming recipe (trusted clicks, arrow wiggles, `Page.bringToFront` + long per-page dwell, `_activeImgRefs` inspection) produced **zero decoded images** across pages 2–14. Pages past the first batch also 403 on their direct URL, so they exist *only* as those on-interaction blobs; the blobs are horizontally-mirrored (`scaleX(-1)`, reversible via `sips -f horizontal`) but that's moot when they never decode under automation. Getting a full, ordered chapter would require reverse-engineering AetherCipher directly (rotating obfuscated `security.oby.js` keys, 29×SHA-256 PoW headers, CSRF/challenge tokens) — the exact stack that got the dedicated Tachiyomi/keiyoushi extension permanently delisted — or a fragile manual babysitting flow. Not worth it; left unsupported. The reusable takeaways: **CF clearance via attach-to-a-user-launched-Chrome** (`chromedp.NewRemoteAllocator` + `page.BringToFront()`, since attached tabs are backgrounded) is a real technique for "works in my Chrome, not in automated Chrome"; and CDP `Input.dispatch*Event` gives isTrusted:true events that drive reader gestures a page's JS refuses from synthetic events — sakura just gates decoding on human-like interaction our automation couldn't fake.
- **An open API doesn't mean there's anything to download.** comick.dev/api.comick.dev sits behind Cloudflare, but a visible browser clears the challenge and the harvested cookies do carry over to the `api.` subdomain (no code changes needed — `http/session.go`'s cookie matching is already domain-suffix based), giving plain-HTTP access to comic/chapter-list JSON. However the actual page-image endpoint (`/chapter/{hid}/get_images`) returns an empty array `[]` anonymously for every chapter tried, from an obscure webtoon to a hugely popular, officially-scanlated one (Solo Leveling) — and the live rendered reader page itself never receives any page images either, matching manual browsing. Not supportable without an account; left unimplemented.
- **atsu.moe** (Atsumaru) is a React SPA (near-empty HTML shell, Next-style client routing) but its JSON api is completely open to plain HTTP, no cookies/session needed: `/api/manga/info?mangaId={id}` returns the title and the *full* chapters list (across every scanlation group) in one uncapped call — no pagination like mangafire/qimanga. Multiple scanlation groups (`scanId`) can each upload the same chapter numbers; the api always mixes them together (there's no per-group filter param), so `atsumaru.go` picks the group with the most chapters and drops the rest to keep numbers unique. Reader pages are `/api/read/chapter?mangaId={id}&chapterId={chapterId}`, and page images are plain AVIF files under `/static/pages/...`, directly downloadable with no referer/auth. Discovering this only took following the network log (`PROBE_NETLOG=1`) on the rendered series/reader pages — grepping the JS bundles for literal endpoint strings came up empty because they're built from a typed api-client object, not string concatenation. Also: Go's `net/http.DetectContentType` doesn't recognise AVIF, so `packer/raw.go`'s `extFromContent` sniffs the ISOBMFF `ftyp` box brand manually to name pages `.avif` instead of mislabeling them `.jpg`.
- **gdscans.com (GalaxyDegenScans) is the same Madara/wp-manga theme `tcb.go` already handles** (chapters via `POST {base}/manga/{slug}/ajax/chapters`), but it groups chapters under "Volume N" `<li>` wrappers — a plain `body.Find("li")` (as `tcb.go` originally did) also matched those volume-header `<li>`s and their empty list-wrapper `<li>`s as bogus duplicate chapters (visible as a spurious `v2` CBZ for the same chapter number). Fixed by scoping to `li.wp-manga-chapter`; lhtranslation.net has a flat chapter list so it was unaffected either way.
- **lagoonscans.com** is plain HTML (`PlainHTML`, `#chapterlist li` rows, `.chapternum` text, `ts_reader.run(...)` images already handled by `getPlainHTMLImageURL`) — it runs Themesia's commercial "MangaReader" WP theme, the same publisher as sushiscan's `ts_reader` theme, no Cloudflare. One long-running series (`solo-max-level-newbie`, 259 chapters) oddly only exposes its latest 6 chapters both in the static HTML and via the theme's own `action=get_chapters` ajax endpoint (used for the reader's chapter-jump dropdown) — a per-series data anomaly, not a site-wide cap: a shorter series (`crimson-reset`, 52 chapters) lists all of them fine and was used for testing/smoke-test instead.
- **The same theme can be plain-HTTP on one domain and Cloudflare-gated on another.** rokaricomics.com uses the exact mangastream/themesia markup already registered for the browser-only sushiscan.net (`h1.entry-title`, `#chapterlist li`, `.chapternum`, `#readerarea img`, `ts_reader.run`), but answers plain HTTP with no challenge — added to `plainhtml.go` instead of `plainhtmlbrowser.go`. Its most recent chapter(s) can be paywalled behind a coin system (reader HTML has a `lock-container`/`lock-title` and no page images); test with an older, unlocked chapter instead of the latest one.
- **sanascans.com (July 2026)**: the plain `sanascan.com` (no "s") domain is an unrelated parked-domain spam page — the real site is `sanascans.com`. It's server-rendered Astro (same island-hydration template family as asurascans.com: `data-astro-prefetch` anchors, tailwind-only classes with no site-specific hooks), so plain HTTP gets the full chapter list and reader images with no browser. Its series page has several decorative `<h1>` stat labels (Status/Type/Chapters/Last update) besides the real title, so a bare `h1` selector concatenates all of them — scope it to `h1[itemprop="name"]`. Chapter rows are `a[data-astro-prefetch][href*="/chapter-"]` (hyphenated `/chapter-14`, unlike asurascans' slashed `/chapter/14`, so the two selectors don't collide despite the shared template). Reader images carry a `data-reader-page-image` attribute. The site runs a coins/premium paywall: newest chapters are often locked (detectable server-side via a `lockedChapter` JSON blob embedded in an `astro-island` `props` attribute) and render with zero page images — no error, just an empty chapter — so pick an older, unlocked chapter when testing or downloading recent-but-locked content.
- **writerscans.com (Writers' Scans)** is plain HTML end to end (chapter list is static, no Cloudflare, no browser needed), but the reader's `<img>` tags only carry a `/assets/images/placeholder.svg` `src` plus an opaque `uid` attribute; the real image lives at a URL an inline `<script>` builds from a template literal (`` `https://cdn.meowing.org/uploads/${uid}` ``). `getPlainHTMLImageURL()` now has a generic fallback for this: it regexes the page's own script for a `` `<prefix>${uid}` `` literal and, if found, builds each image URL from `prefix + uid` instead of reading `src`/`data-src` — reusable for any other site sharing this lazy-load pattern.
- **toonily.com** is a standard Madara theme behind a Cloudflare managed challenge — same selectors as toongod/dragontea/manhuaus, just add the domain to that existing `BrowserSiteSelector` entry. One gotcha: its series URLs are `/serie/{slug}-{hash}/`, not the more common Madara `/manga/{slug}/` — a guessed `/manga/solo-leveling/` URL never returned a real page (curl only ever saw the CF challenge, and one rendering attempt ended up on a bare cover-image page instead), so don't guess the slug pattern; get the real series URL from the homepage/browse listing (`div.page-item-detail a`) first.
- **A virtualized/infinite-scroll reader isn't a dead end either.** comix.to's reader only mounts a handful of `<img>`s near the top; a plain `GetHTML` snapshot missed the rest. `browser.GetHTMLWithScroll(url, waitSelector, iterations, pause, timeout)` scrolls in fractional steps of the page's current (growing) `scrollHeight` after the wait selector matches, letting an IntersectionObserver-driven reader progressively mount everything before the final HTML is read — no need to know the real page count up front, and a fixed ~35 iterations at 600ms comfortably covers chapters of very different lengths since each step re-reads the live scrollHeight.
- **A CF challenge that keeps timing out on a rich selector may still pass in seconds — the selector is just too specific too early.** drakecomic.org (Drake Scans, a `mangareader`-themed WordPress/themesia site, same family as sushiscan.net: `#chapterlist li`/`.chapternum`/`ts_reader.run(...)`) repeatedly failed `browser.GetHTML` waiting on `li.wp-manga-chapter`/`#chapterlist li` (up to 4 minutes, visible browser) even though `PROBE_NETLOG` showed the real page answering 200 within seconds of the challenge — the page's `<body>` genuinely renders empty at first and only gets populated a few seconds later by an `admin-ajax.php` call. `chromedp.WaitVisible` already polls until the selector appears (no extra settle needed), so this wasn't actually a dead end; waiting on `"body"` with a probe `PROBE_SLEEP` first, then diffing the dump for the real selectors, unblocked it. Its images (`wp-content/uploads/.../*.webp`) are also CF-protected (403 to plain curl) but download fine via the harvested browser session, same as any other Path C site.
- **A blob in the DOM that isn't comma-separated URLs isn't necessarily unbreakable.** leercapitulo.co's reader hides page images in an obfuscated `#array_data` string that plain HTTP/regex can't decode (unlike mangabats' plain `var chapImages = '...,...'`). But the site's own javascript decodes it fine — the problem was it only showed one page at a time by default. Rendering the page in a real browser (`browser.GetHTML`) revealed a `localStorage` flag (`display_mode`) that switches the reader to "load all pages at once"; setting it and reloading before scraping got every page's real URL with zero cipher reverse-engineering. `browser.GetHTMLWithLocalStorage` generalizes this (set key/value, reload, then wait/extract) for any site gating content behind a client-side preference. Before assuming a reader's images are "encrypted and unsupportable" like colamanga, check whether a browser alone (no interaction) already resolves them — colamanga's are genuinely client-side-encrypted even after rendering, this one wasn't.
- **An obfuscated-looking JS array is still plain data if it's not actually encrypted.** mangakatana.com's reader `<img>` tags only ever carry a `data-src="#"` placeholder; the real per-page URLs live in a plain JS array assigned client-side (`obj.attr('data-src', thzq[i])`) whose variable name (`thzq`) isn't fixed across page loads/deployments. `getPlainHTMLImageURL` finds the name from that assignment call first, then pulls the array literal by name — no headless rendering or cipher-reversing needed, it's just string data sitting in the raw HTML.

- **demonicscans.org is fully plain HTTP, no Cloudflare.** Chapter list is server-rendered (`#chapters-list li`, link/number in the `a.chplinks`), and the reader page embeds full-size, non-lazy image URLs directly (`img.imgholder`, no JS variable to unpack). The only wrinkle: its CDN (`cdn.demoniclibs.com`) serves image URLs with literal, unencoded spaces in the path (e.g. `/Return of the Mount Hua Sect/172/0.jpg`) — Go's `net/http.NewRequest` percent-encodes these automatically, so no special handling was needed. Also prompted a small general fix: `PlainHTML.FetchChapters()` now runs `sanitizeTitle()` on the chapter title (previously only the series title was sanitized), since demonicscans' chapter rows mix the chapter text with a trailing date `<span>` and produced titles full of raw whitespace.
- **ritharscans.com** needs no browser at all: chapter list is a static `#chapters a` list, and the reader (an Alpine.js/Livewire app) inlines every page's relative path plus a `baseLink` prefix directly in an `x-data="immersiveReader({ pages: [...], baseLink: '...' })"` HTML attribute — goquery decodes the `&quot;`-escaped JSON transparently via `.Attr()`, no cipher or JS execution needed (see the new branch in `getPlainHTMLImageURL`). Recent chapters can be coin-gated (a "Coin"/VIP badge on the chapter row); test with an older, free chapter.
- **Madara's `div.reading-content img` isn't always clean.** manhuatop.org (July 2026) is a standard Madara theme (`li.wp-manga-chapter`, `div.post-title h1`) matching the existing toongod/manhuaus browser entry's selectors, but its reader also throws in a decorative `about_manhuatop_*.webp` banner and a `To_be_continued.webp` footer sharing the same `wp-manga-chapter-img` class as the real, sequentially-`id`'d (`image-0`, `image-1`, ...) page images — a plain `div.reading-content img` selector would smuggle those two junk images in as fake first/last pages. Its own dedicated browser-selector entry uses `div.reading-content img[id^="image-"]` instead. Images download via plain HTTP with no cookies needed once the browser harvests the session.
- **A Cloudflare 403 on one route doesn't mean the whole site is CF-blocked — check whether it's only a direct-navigation rule.** mkissa.to (see `mkissa.go`) is a SvelteKit SPA over a GraphQL API (Apollo persisted queries, `api.mkissa.net/api?variables=...&extensions={"persistedQuery":{"sha256Hash":...}}`): the manga-info query (title + the full per-chapter-number list, in `data.manga.availableChaptersDetail.sub`) answers plain HTTP with zero cookies. But `/manga/{id}/chapter-{n}-sub` (the reader) 403s with a Turnstile challenge on *any* direct top-level navigation — even a chromedp session that first visited the series page and carried its cookies into a second `Navigate()` to the reader URL still gets blocked, ruling out a simple cookie-warmup fix. The reader loads fine, though, when reached via the app's own client-side routing: render the series page, click the "Chapters" tab button, then click the specific chapter's `div[data-href="/manga/{id}/chapter-{n}-sub"]` (older chapters are paginated behind `.media-ep-list__pagination button`s that must be clicked through first) — this never issues a real navigation request, so the WAF rule never fires. The reader also "encrypts" its page list (a `tobeparsed` blob behind a signed `aaReq` token, decoded only by their own JS, same idea as sakuramangas' AetherCipher) but that's moot: the decoded `<img class="reader-page__img" src=...>` elements are lazy-loaded plain JPEG/PNG URLs on a third-party CDN (`ytimgf.youtube-anime.com/images9/{id}/{num}/sub_{ts}/{page}.jpg`), directly downloadable with a plain UA+Referer, no cookies needed. Getting every page requires scrolling: only ~2 images resolve immediately, the rest need repeated `scrollBy(0, innerHeight)` + settle, and some mkissa readers auto-continue into the *next* chapter once you scroll past the last page (appending more `.reader-page` elements to the same DOM) — so stop scrolling on a same-count plateau of images whose `src` contains `/{id}/{num}/`, not on a fixed total or a generic selector count. `browser.GetReaderHTML` generalizes this click-through-then-plateau-scroll pattern for any SPA whose reader route is blocked on direct navigation but reachable via in-app routing.

### Adding support for a new site

- If the site is plain HTML (chapter list + images in the reader page): add a `SiteSelector` entry to the list in `PlainHTML.Test()`, and add a smoke-test URL to the `grabber/html` makefile target and the README's supported sites list.
- Otherwise: create a new grabber implementing `Site` (embed `*Grabber`), and register it in `IdentifySite()`.

### Investigating/testing sites

Lessons from past site-support work (July 2026):

- **Triage with curl before touching Go.** Fetch the manga page with a browser User-Agent and check the `<title>`: "Just a moment..." means a Cloudflare JS challenge and the site cannot be supported with plain HTTP (same for TLS-level connection drops, empty-`<div id="app-root">` SPAs, or encrypted-image readers like colamanga). Don't burn time debugging the grabber in those cases.
- **Verify CBZ contents, not just their existence.** A run can "succeed" while producing empty or junk archives. Check that entries have non-zero sizes and real image magic bytes (JPG/PNG/WebP/GIF).
- **403/500 in the app but 200 in curl almost always means headers.** `http/request.go` sends a browser User-Agent, Accept, Accept-Language, and referers with a trailing slash — all four exist because some site rejected requests without them. Compare headers before assuming a site is down.
- **Smoke-test with recent chapters.** Some sites drop old content from their CDNs (inmanga 404s on early One Piece chapters) and MangaDex replaces licensed chapters with pageless external stubs, so "chapter 1" is often the worst test target.
- **Sites move domains constantly.** Before declaring a site dead, probe for successors (e.g. tcbscans.com → tcbonepiecechapters.com, mangabat → mangabats.com, asuratoon → asurascans.com); redirects may only preserve the homepage, not series paths.
- The makefile is tracked as lowercase `makefile`: on macOS's case-insensitive filesystem editing `Makefile` works, but `git add` needs the lowercase name.
- macOS has no `timeout` command; use the Bash tool's timeout or a background-kill wrapper when running live-site smoke tests.
- **A near-empty `grep -o '<img src="..."'` result doesn't mean the images are JS-loaded.** rawkuma.net's reader page (Kiru WordPress theme, Path A) embeds every page as a plain `<img src='...'>` in the raw HTML, but with single-quoted attributes — a double-quote-only grep pattern misses them entirely (goquery parses them fine regardless). Confirm with a real HTML parser, or a quote-agnostic grep, before concluding a reader needs a browser. Also: a bare `h1` selector can be ambiguous even on a "real HTML" site — rawkuma's series page has a second, earlier `<h1>` for an unrelated "Last Updates" sidebar label, so `Title` had to be scoped (`h1[itemprop="name"]`); `grep -c` on a single-line minified HTML file counts matching *lines*, not occurrences, so it can hide a second match — use `grep -o | wc -l` instead.
- When multiple agents investigate sites in parallel on the same machine, generic scratch paths like `/tmp/series.html` or `/tmp/reader.html` collide across agents/worktrees and get silently overwritten mid-investigation (a file can shrink or change content between two of your own commands with no edit of your own). Use a unique temp dir per investigation, e.g. `/tmp/agent-<site>/`.
- **A Cloudflare-looking cookie wall isn't always Cloudflare.** fanfox.net (Manga Fox) returns real, complete HTML (chapter list, JS vars) with plain HTTP/curl — no CF challenge, no TLS block — but every series page, even for a completely non-mature title like Chainsaw Man, is replaced by an age-confirmation interstitial (0 chapter rows) unless a plain `isAdult=1` cookie is sent; `http.SetCookie("fanfox.net", "isAdult", "1")` in `Test()` is enough, no browser needed. Always check response *content* (chapter rows present?), not just HTTP status/size, before escalating to Path C.
- **A "packed" JS blob isn't encryption either.** fanfox's reader resolves each page's image through `chapterfun.ashx?cid=...&page=N`, whose response is obfuscated with the 20-year-old Dean Edwards "P.A.C.K.E.R." format (`eval(function(p,a,c,k,e,d){...}(payload,radix,count,dict.split('|'),0,{}))`) — a deterministic, publicly-documented base-N token substitution, not a cipher. It unpacks completely in pure Go with zero JS engine/dependency (see `unpackPackerJS` in `fanfox.go`). Don't reach for a browser or a JS VM just because a response starts with `eval(function(p,a,c,k,e,d)`.
- **When probing in parallel with other agents on the same machine, don't use predictable `/tmp` filenames** (`/tmp/series.html` etc.) — a sibling agent's probe can silently overwrite yours mid-investigation. Use a directory namespaced to your own worktree/agent id.
- en-thunderscans.com is a plain mangastream/themesia theme (`ts_reader.run`, already handled generically) but coin-locks recent chapters behind a modal (`data-bs-toggle="modal"`, no `href`) instead of a real link; a `Rows` selector ending in `a[href]` filters those out of the chapter list cleanly instead of yielding chapters with bogus (base-URL) download links. Also found (and fixed) that `PlainHTML.FetchChapters` never ran chapter titles through `sanitizeTitle`, so themes with heavily-indented chapter-number markup leaked raw tabs/newlines into filenames.
- witchscans.com is a plain-HTTP mangastream/themesia WordPress theme (`#chapterlist li` rows, `.chapternum` text, `ts_reader.run` reader blob already handled by `getPlainHTMLImageURL`) — no browser needed. Its `.chapternum` span text is padded with a lot of internal whitespace/newlines from the template (`"Chapter                            600     "`), which `parseChapterNumber`'s `\s*` tolerates fine but left the raw whitespace in the chapter title; `FetchChapters` now runs the chosen title through `sanitizeTitle()` before storing it, fixing this for every `PlainHTML` site, not just this one.
- **mistscans.com** (July 2026, former Arven Scans members) is plain HTML, no Cloudflare, no browser needed. Its reader lazy-loads pages by giving each `<img>` a placeholder `src` plus a `uid` attribute, then building the real URL client-side from a JS template literal (`` `https://cdn.meowing.org/uploads/${uid}` ``). `getPlainHTMLImageURL()` now extracts that literal's prefix straight out of the page's own `<script>` (`uidTemplateRe`) instead of hardcoding the CDN host, so the same fallback should work for any other site built on this template even if its CDN domain differs. Also note: newly-released chapters are coin/paywall-gated for a few days (a lock icon + coin badge in the chapter grid) — smoke-test with a chapter a bit behind the latest, not the newest one.
